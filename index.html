VectrDraw vectors. Get React code.

Vectr is a design-to-code tool that eliminates the gap between what a designer draws and what a developer can use. Every vector drawn on the canvas is simultaneously clean, readable, component-ready React and SVG code.

The tool has two tabs — a Design tab (a canvas like Illustrator) and a Code tab (live React output) — with an optional split view. An AI assistant lives inside, aware of the current code at all times.

Under the hood, two machine learning models clean the strokes as they're drawn and decompose complex shapes into named, reusable components. A third AI model helps developers understand and modify the generated code in plain language.

01 — The Problem

Design and code don't speak the same language

When a designer draws a custom vector or complex shape, a developer has no good way to turn that into usable code. The design loses its soul in translation — and today's tools make this worse, not better.

The designer

Makes something beautiful

Custom shapes, detailed paths, intricate illustrations — drawn with precision, built to be scalable, animatable, and alive. Illustrator, Figma, Sketch. They know exactly what they want and they produced it. The vector is perfect.

The developer

Gets a PNG and guesses

No scalability. No interactivity. No way to animate it, theme it, or respond to events. Alternatively they get a 400-line bloated SVG dump from Illustrator that nobody wants to touch — full of transform matrices and nested groups with no meaningful structure.

What actually happens today — and why it fails
🖼

Export as PNG

Loses all scalability. Can't be animated. Can't change color without a new export. Not a living part of the codebase — just a dead image file.

📄

Export SVG from Illustrator

Produces hundreds of lines of unreadable, unstructured SVG. No component structure. Impossible to modify programmatically. Developers copy-paste it as a black box and pray.

🔁

Manual recreation by developer

The developer hand-rewrites the shape in SVG or Canvas. Takes hours. Introduces discrepancies. Requires going back and forth with the designer. Not sustainable at scale.

✓

Vectr's approach

The vector is the code from the moment it's drawn. The designer doesn't touch code. The developer doesn't touch the canvas. Both see the same thing in their own language, in real time.

03 — Intelligence Layer — SC Contribution

Two ML models, one AI assistant

One model cleans what comes in. Another structures what goes out. An AI assistant helps the developer work with what's been generated. Together they form the scientific computing core of the tool.

01

Stroke to Clean Path Model

Runs in real time as the designer draws. Cleans raw freehand input into minimal, correct SVG paths.

SC — Model 1

What it does

When a designer draws freehand, the raw input is messy — shaky lines, uneven curves, hundreds of redundant points from every pixel the mouse touched. This model takes that raw stroke data and outputs a clean, minimal SVG path in real time. Think of it as autocorrect for vectors — it smooths, simplifies, and corrects without the designer noticing anything changed.

How it's trained

Trained on paired datasets: raw hand-drawn stroke input on one side, the corresponding clean SVG path on the other. The model learns the mapping between the messy real-world input and the ideal vector output. It operates with low latency — it must run fast enough to feel instant as the designer draws, so inference speed is a core technical constraint.

02

Vector Shape Decomposition Model

Analyzes completed vectors and breaks them into named primitive components — circles, bezier curves, straight lines.

SC — Model 2

What it does

When a designer draws something complex — a logo, an icon, a detailed illustration — this model analyzes the full vector and classifies each segment: which parts are circles, which are bezier curves, which are straight lines, which are compound paths. Each decomposed part becomes a named, reusable piece in the generated React code. Instead of one giant unreadable SVG path, the output is structured and segmented.

The library gatekeeper

This model also acts as the gatekeeper for the shared library. When a designer uploads a finished vector, this model runs on it. If it can fully decompose the vector into understood primitives, it becomes editable in the library — anyone can drag it onto the canvas and modify individual parts. If it can't, the vector uploads as locked and usable as-is. The better the model gets over time, the more the library opens up.

AI

Context-Aware AI Assistant

Lives inside the tool with full awareness of the current code state. Developers talk to it in plain language.

SC — AI Layer

What it does

The assistant is not a generic chatbot. It reads the live code panel at all times and responds specifically to what's on screen. A developer can ask it what a path does, ask it to make a shape responsive, restructure a path into a reusable component, or explain why a bezier curve is behaving unexpectedly. It understands SVG structure, React component anatomy, and design-to-code translation.

Simulation before application

Before applying any code change, the assistant simulates how the change would affect the rendered output and shows a diff. This connects to modeling and simulation — the assistant models the effect of its own suggestion before it makes it. The developer approves or rejects. Nothing is applied blindly. This makes it safe to use in a real project without reviewing every token it outputs.

04 — CS Contribution

The engineering underneath

The CS team builds the system that the ML models live inside — the canvas, the code generation pipeline, the architecture, and the export. This is where compiler theory, system design, and computer graphics come together.

CS — 01

Canvas & Rendering Engine

Building the design canvas itself — the pen tool, shape tools, bezier curve manipulation, anchor point handles, layers panel, selection, transformations. This is a core computer graphics and software engineering problem involving real-time rendering, complex event handling (mouse, touch, keyboard shortcuts), and managing a deep canvas state tree that the rest of the system reads from.

The canvas state is the single source of truth — every other part of the system (code generation, ML models, the assistant) reads from it. Getting this state model right is the most critical architectural decision in the CS contribution.

Computer Graphics Software Engineering Data Structures
CS — 02

Code Generation Pipeline

The system that takes the canvas state (stored as structured JSON) and converts it to clean React and SVG code in real time. This involves designing the data structures that represent each shape, the translation logic that converts each shape type to its corresponding React component, and ensuring the output is human-readable and well-organized — not just correct.

This is a compiler theory problem at its core. The visual language of the canvas is the source language. React code is the target. The pipeline is a compiler with a custom intermediate representation in between.

Compiler Theory Concepts of Programming Languages Analysis & Algorithms
CS — 03

Architecture & Infrastructure

The overall system design — how the frontend, backend, database, autosave system, and real-time sync all connect and communicate. This includes designing the API layer that the frontend calls to save projects and retrieve the shared library, handling concurrent edits if multiple users are in the same workspace, and ensuring the system is resilient and performant under load.

Security matters: projects contain proprietary design work. User authentication, encrypted storage, and secure API communication are non-negotiable requirements, not afterthoughts.

Operating Systems Computer Networks Distributed Computing Cybersecurity System Analysis & Design
CS — 04

Export Tool & Data Mining

The export tool packages the generated React/SVG code into clean, self-contained component files — ready to drop into any project with a single import. This involves file system operations, code formatting, dependency resolution, and ensuring the exported file is always optimized and readable without any runtime dependency on Vectr itself.

The data mining layer analyzes usage patterns across the tool — what shapes designers draw most, how paths are structured, what the assistant gets asked — feeding training data back to the SC models and informing future feature decisions through pattern recognition.

05 — The Vector Library

A shared library that gets smarter over time

Designers can upload finished vectors to a shared library that the whole team can use. The Shape Decomposition model runs on each upload — determining whether the vector becomes editable or stays locked, and improving that split as it improves.

Editable vectors

When Model 2 can fully decompose an uploaded vector into understood primitives, it becomes editable in the library. Anyone can drag it onto the canvas, click into it, and modify individual sub-components — a circle's radius, a bezier curve's handle, a line's weight. Each part is a named, selectable React component.

Locked vectors

When Model 2 can't fully understand the structure of an upload — too complex, too unusual, or containing patterns it hasn't been trained on — the vector uploads as locked. It's still usable, droppable onto the canvas, and exportable as-is. But it can't be broken apart. As the model improves with more training data, previously locked vectors can be re-evaluated and unlocked automatically.

User-assisted decomposition

Help the model unlock your vector

If a vector is locked but you still need its code, you can assist the model manually. The tool highlights the unrecognized regions and lets you fill each one by selecting a matching primitive from the app's built-in shape library — a circle, a bezier curve, a rectangle. Once you map the unknowns, the model has enough context to decompose the full vector, generate clean code for it, and mark it editable. Your mapping also gets used as new training data, making the model better for everyone going forward.
