VectrDraw vectors. Get React code.

Vectr is a design-to-code tool that closes the gap between what a designer draws and what a developer can actually use. Every vector drawn on the canvas is simultaneously clean, readable, component-ready React and SVG code.

The tool has two tabs — a Design tab (a canvas like Illustrator) and a Code tab (live React output) — with an optional split view. An AI assistant lives inside the tool, always aware of the current code state, so developers can ask it questions or request changes in plain language.

This is a pure CS project. The canvas engine, the code generation pipeline, the architecture, the AI integration, the export system — all of it is built from the ground up and maps directly to the CS curriculum.

01 — The Problem

Design and code don't speak the same language

When a designer draws a custom vector or complex shape, a developer has no good way to turn it into usable code. The design loses its soul in translation — and today's tools make this worse, not better.

The designer

Makes something beautiful

Custom shapes, detailed paths, intricate illustrations — drawn with precision, built to be scalable, animatable, and alive. They work in Illustrator or Figma. They know exactly what they want and they produced it. The vector is perfect.

The developer

Gets a PNG and guesses

No scalability. No interactivity. No way to animate it, theme it, or respond to events. Or they get a 400-line SVG export from Illustrator that nobody wants to touch — full of transform matrices and meaningless auto-generated IDs with no component structure at all.

What happens today — and why it fails
🖼

Export as PNG

Loses all scalability. Can't be animated or themed. Not a living part of the codebase — just a static image file that breaks on retina displays and can't respond to anything.

📄

Export SVG from Illustrator

Produces hundreds of lines of unreadable SVG. No component structure. Impossible to modify programmatically. Developers copy-paste it as a black box and nobody touches it again.

🔁

Developer hand-rewrites the shape

Takes hours. Introduces discrepancies. Requires constant back-and-forth with the designer. The developer has to guess at bezier control handles from looking at a PNG. Not sustainable at any scale.

✓

Vectr's approach

The vector is the code from the moment it's drawn. The designer works on the canvas. The developer works in the code tab. Both see the same thing in their own language, in real time, with no translation step in between.

02 — The Tool

Two tabs. One source of truth.

A design tool and a code editor living in the same workspace. Every change on the canvas is immediately reflected in the code, whether you're looking at it or not. The code is always clean, always current, always exportable.

01

Design Tab — the canvas

A canvas that feels like Illustrator. Designers work the way they already know how — draw, adjust, refine. No code knowledge required. The tool handles everything underneath.

  • Pen tool with full bezier curve support — place anchor points, drag control handles, close paths, edit existing nodes
  • Shape tools — rectangle, circle, ellipse, polygon, freehand path
  • Fill, stroke, opacity, layer management, grouping
  • Pre-built vector library to drag and start from
  • UI component primitives — buttons, navbars, inputs — editable on canvas

Everything drawn on the canvas is reflected here instantly as clean, readable React and SVG. The code updates in the background whether or not you're looking at it. Developers can read it, copy from it, or ask the AI assistant to work with it.

  • Each shape gets its own component with named props (fill, stroke, size, className)
  • Component hierarchy mirrors visual grouping on the canvas

  • Syntax highlighted, formatted, and readable by a human

03 — CS Contribution

The engineering underneath

Vectr is built entirely by the CS team. The canvas, the code generation pipeline, the architecture, the AI integration, and the export tool all map directly to core CS courses — not abstractly, but concretely.

CS — 01

Canvas & Rendering Engine

This is the entire design canvas — the pen tool, shape tools, bezier curve manipulation, anchor point handles, layers, selection, and transformations. It's built using Paper.js, a vector-native canvas library that gives us direct access to paths, curves, and scene graph management.

Every element on the canvas — a rectangle, a bezier path, a circle — has a live internal representation as structured data: anchor positions, control handle positions, fill color, stroke, opacity, z-index. This is the canvas state tree, and it's the single source of truth that every other part of the system reads from. Getting this right is the most critical architectural decision in the whole project.

Managing all user input events correctly (mouse, touch, keyboard shortcuts), redrawing efficiently on every frame, and handling selection, drag, and resize without visual glitches — this is a core computer graphics and real-time rendering problem.

Computer Graphics Software Engineering Data Structures
CS — 02

Code Generation Pipeline

This is the core innovation of the project. At any given moment, the canvas state is a structured JSON object — a complete description of every shape and path with all their properties. The code generation pipeline reads that JSON and produces clean, readable React components in real time.

Think of this as a compiler where the source language is the visual canvas and the target language is React/JSX. Every shape type has its own generator function. A rectangle becomes a component with named variables for x, y, width, height, radius, fill, and opacity. A bezier path becomes a component where every anchor point and every control handle is a named variable — not a raw coordinate dump. The output is code a developer can actually read, understand, and modify.

The pipeline also handles grouping, nesting, naming conventions, and ordering consistently — which is an algorithms problem as much as it is a language design problem.

Compiler Theory Concepts of Programming Languages Analysis & Algorithms
CS — 03

Architecture & Infrastructure

The full system design — how every part of the application connects and communicates. The frontend is React. The backend is Node.js, handling API requests, user authentication, project storage, and the autosave system. The database stores projects as JSON snapshots of the canvas state, which the frontend can reload and reconstruct exactly.

The autosave system triggers every 5 minutes and on every Ctrl+S. Because running a full code generation pass on every single keystroke would be too expensive, the save system is decoupled from the live code generation — generation runs continuously but saves are batched and queued.

Security is not optional: projects contain proprietary design work. User authentication, encrypted project storage, and a backend API key proxy (so the AI key is never exposed to the client) are hard requirements. The distributed side covers how the system handles multiple users accessing the shared library simultaneously without conflicts.

Operating Systems Computer Networks Distributed Computing Cybersecurity System Analysis & Design
CS — 04

Export Tool & Data Mining

The export tool takes the generated React/SVG code and packages it into a clean, self-contained component file — formatted, named correctly, and ready to drop into any project with a single import. No cleanup. No runtime dependency on Vectr itself. This involves file system operations, code formatting, dependency resolution, and ensuring the output is always optimized and human-readable.

The data mining layer tracks usage patterns across the tool — which shapes get drawn most, how paths are typically structured, what users ask the AI assistant most often. This feeds directly into training data improvement and informs future feature decisions. Pattern recognition identifies recurring shape types and helps surface the most useful vectors in the shared library.

04 — AI Assistant

A developer-facing AI that knows what's on screen

The AI assistant is not a generic chatbot dropped into the sidebar. It's specifically integrated into the code state and responds to exactly what's currently on screen — giving developers a natural language interface to the generated code.

AI

Context-Aware Code Assistant

Lives inside the tool. Reads the live code panel on every message. Understands SVG structure, React anatomy, and the naming conventions the code generator produces.

CS — NLP / Neural Networks

What it does

With every message, the assistant receives the full current component code as context. It's not a generic chatbot — it's specifically prompted to understand SVG structure, React component anatomy, and the naming conventions the code generator produces.

A developer can ask it what a control point does, ask it to make a shape responsive, turn a path into a reusable component with props, or animate a variable using Framer Motion. The assistant always knows exactly what's on screen.

How the integration works

The live code state is injected into every API call as a system context. The response is parsed and rendered in the assistant panel. When the assistant suggests a code change, the suggested diff is shown to the developer for review — nothing is applied without confirmation.

The AI API key lives on the backend and is never exposed to the client. All requests are proxied through the Node.js server, which also handles rate limiting and request logging.

05 — The Vector Library

A shared library your team builds over time

Designers can upload finished vectors to a shared library that the whole team can drag onto any canvas and start from. Each library item stores both the canvas JSON and the generated code — so everything is re-editable, not just a static file.

How library items work

When a designer uploads a finished vector, two things get stored: the canvas JSON (a complete structured description of every anchor point, curve, and property) and the generated React code. Either can be loaded back independently.

When someone pulls a library item onto the canvas, the tool reconstructs the full editable vector from the stored JSON — all the anchor points come back, the bezier handles are live, and the code panel updates immediately to match. It's not a static import; it's a full reconstruction.

Building the library over time

The shared library grows as the team uploads their best components and vectors. Frequently used shapes — a brand icon, a recurring UI element, a complex illustration that took hours to get right — get stored once and reused everywhere.

The data mining system tracks which library items get pulled onto the canvas most, which ones get modified immediately after being dragged in, and which ones get exported directly. Over time this creates a clear picture of what's actually useful, which feeds back into how the library is organized and surfaced.
